《服药宝》APP - 产品需求文档（PRD）

版本： v2.7 MVP（最终定稿）

日期： 2026年3月4日

状态： 待开发

核心目标： 8.5周内上线一款以基础提醒为核心、同时满足事件驱动用药场景和用户停用需求的App，验证用户需求并形成差异化竞争力。



1\. 产品概述

1.1 产品背景

用药依从性差是慢性病管理中的普遍问题，尤其对于需要长期服药的人群（如高血压、糖尿病、HIV感染者），忘记服药可能导致病情反复甚至加重。同时，部分用户（如HIV PrEP使用者）需要根据事件（如性行为）临时调整用药方案，现有固定周期提醒无法满足。老年人群体对手机操作不熟悉，更需要简单、清晰的用药提醒工具。



1.2 产品定位

一款专注于用药/保健品提醒的工具类App，核心价值在于帮助用户准时服药，同时率先推出简化版事件驱动（多天用药计划）功能，满足特定人群的按需服药需求。此外，提供停用机制，让用户可灵活管理不再服用的项目而不丢失历史记录。兼顾适老化设计（大字、语音），为后续商业化迭代打基础。



1.3 目标用户

用户类型	描述

核心用户	需要长期规律用药的人群（慢性病患者、HIV感染者）

次核心用户	有日常保健品服用需求的用户（维生素、鱼油等）

特色用户	HIV PrEP使用者、紧急避孕药使用者、短期抗生素用户等需要事件驱动用药的人群

4\. 核心功能详述

4.1 药品/保健品管理

4.1.1 添加方式

内置药品库：200种常用药，用户通过搜索选择。选中后自动填充常见用法用量（如”每日一次，每次一片，餐后”），用户可修改。

**药品库数据结构（JSON格式）：**
```json
{
  “name”: “药品名称”,
  “category”: “medicine”,  // 或 “supplement”
  “defaultDosage”: “1片”,
  “defaultUsage”: “餐后”
}
```



手动添加：支持完全手动输入药品名称、剂量、用法。



新增“多天用药计划”选项：在添加页增加开关或标签，用户可选择创建多天计划。



4.1.2 多天用药计划

目标：满足事件驱动用药场景的简化需求，允许用户创建连续多天的提醒计划。



交互流程：



用户在添加药品时，开启“多天计划”开关（或选择计划类型）。



设置计划参数：



起始日期（默认今天，可修改）



持续天数（1-7天，可增减）



提醒时间（默认当前时间，所有生成的提醒使用同一时间）



每天剂量：可设置为“统一剂量”（所有天数相同）或“逐天设置”（允许每天不同剂量）



若选择“逐天设置”，弹出每日剂量输入界面（如第1天2片，第2-3天各1片）。



点击保存后，系统根据起始日期和持续天数，生成连续多天的提醒，每天一条，时间均与设置的提醒时间一致。



生成的提醒独立存在，与普通提醒行为一致（支持确认、稍后、跳过、重复提醒）。



用户可将该计划保存为模板，模板存储”持续天数”和”每日剂量序列”（如\[2,1,1]），名称由用户自定义。使用时用户需指定药品、起始日期、提醒时间。

**模板管理入口：** 在添加页增加”使用模板”入口，用户可选择已有模板或保存当前设置为新模板。



数据存储：



计划本身不单独存储，而是直接生成多条Medication记录，每条记录通过plan\_group\_id字段关联（同一计划的记录共享同一个ID）。



停用行为：**当前版本仅支持单条停用，不支持批量停用整个计划。** 多天计划的每条记录独立停用，用户需逐条操作。未来版本可扩展”停用整个计划”功能。



4.1.3 连续添加模式

用户保存一个项目后，弹出浮层询问“是否继续添加同一时间的其他项目？”。



点击【继续添加】进入添加页，时间默认填充为刚才保存的时间（可修改）。



可循环操作，直至点击【完成】。



4.1.4 停用机制

目标：允许用户停止提醒但不删除历史记录，便于未来恢复。



停用操作：



在活跃项目的详情页底部，放置“停用”按钮（红色文字）。



点击后弹窗确认：“停用后将不再提醒，但历史记录会保留。您可在药箱底部的‘已停用项目’中恢复。是否确认停用？”



确认后，将该项目isActive设为false，stoppedAt记录当前时间。**用户反复停用/恢复时，每次停用操作都会追加一条停用记录。** 项目从活跃列表移至药箱底部的停用列表。



恢复操作：



在停用项目的详情页底部，放置“恢复”按钮（绿色文字）。



点击后弹窗确认：“恢复后将重新开始提醒。是否确认恢复？”



确认后，将该项目isActive设为true，stoppedAt记录恢复时间（追加记录）。项目从停用列表移回活跃列表，提醒按原有规则重新生效。



恢复后当天的触发逻辑：**以当前系统时间判断**。若项目有多个提醒时间，**任一提醒时间未过则当天触发**，从次日开始按原有规则触发；若所有时间都已过，则当天不再触发。



停用项目的提醒：停用项目不参与任何提醒调度。



停用项目的历史记录：在记录模块中，停用项目的历史记录仍然正常显示，计入日历圆点和统计。



4.1.5 永久删除

在活跃项目或停用项目的详情页底部（或左滑菜单），提供“永久删除”按钮。



点击后弹窗警告：“此操作将永久删除该项目及其所有历史记录，无法恢复。是否确认？”



确认后，从数据库中彻底删除该项目及其关联的服药记录。



4.1.6 编辑规则

修改提醒时间：若新时间晚于当前，当天按新时间触发；若早于当前，当天不再触发，从次日开始。



修改剂量/用法/名称：仅影响未来提醒，历史记录不变。



修改品类（药品↔保健品）：允许修改，但历史记录中的品类不变。



多天计划编辑：若项目属于多天计划，修改起始日期或持续天数时，需弹窗提示：“此操作将重新生成未来提醒，已发生的记录将保留。是否继续？”确认后，保留已发生的服药记录（已服/跳过），删除未发生的提醒（未来日期），并根据新参数重新生成未来提醒。



4.2 提醒设置

4.2.1 提醒规则类型

每日固定时间



间隔周期（如每8小时）



每周特定几天



每月特定日期



多天计划（一次性）：作为特殊类型，由系统生成多条一次性提醒，不存储为周期规则。



4.2.2 合并提醒

同一时间的多个项目，仅发送一条通知，点击后进入合并清单页。



清单页列出所有待服项目，每个项目有独立确认按钮，底部有“全部已服”按钮。



**完成条件：部分确认后即解除合并，剩余未确认项目独立触发后续重复提醒。**



4.2.3 重复提醒与稍后提醒

重复提醒机制：**从通知发送时刻开始计时**，首次提醒后15分钟未确认 → 第一次重复 → 再过15分钟未确认 → 第二次重复 → 最多重复3次（含首次共4次）。



稍后提醒规则：点击“稍后提醒”后，推迟10分钟再次触发，计入重复次数。



合并提醒场景：在合并清单页点击“稍后提醒”，所有未确认的项目均推迟10分钟。



4.3 提醒形式

项目	说明

通知形式	所有提醒均通过系统本地通知发送，展示在通知栏或横幅（取决于手机系统）。用户点击通知后，进入App对应的确认页面（若为合并提醒，则进入合并清单页）。

声音与震动	系统通知默认带有声音和震动（由手机系统设置决定，App不单独控制）。用户可通过手机系统设置调整通知音量、关闭震动或启用勿扰模式。

语音播报	在满足语音提醒条件时（总开关、对应品类开关、不在免打扰时段），系统将在发送通知的同时，通过TTS引擎播报语音内容。语音播报独立于通知声音，可单独控制。

免打扰时段对通知的影响	免打扰时段（固定24:00-08:00）仅控制语音播报，不控制系统通知的发送。系统通知仍会正常送达，但用户可通过手机系统的勿扰模式自行静音。如需完全静音，用户可在睡前关闭所有语音提醒，并开启手机勿扰模式。

弹窗提示	App内部不主动弹出弹窗，所有提醒依赖系统通知。用户点击通知后进入App界面。

4.4 语音提醒

项目	说明

技术实现	系统TTS引擎（Android TextToSpeech / iOS AVSpeechSynthesizer）

播报内容	“请服用{药品名}，{剂量}”（合并提醒时播报合并内容，如“请服用硝苯地平1片、鱼油1粒”）

触发规则	总开关开启 + 对应品类开关开启 + 不在免打扰时段

免打扰时段	固定为24:00 - 08:00，不可调节

成本	零成本

4.5 数据隐私与安全

措施	说明

数据本地化	所有数据仅存储在手机本地SQLite数据库，绝不传输至云端

数据库加密	使用SQLCipher进行AES-256加密，密钥由Android Keystore / iOS Keychain存储

无账号系统	不收集手机号、邮箱等个人标识信息

无第三方SDK	v2.7不集成任何分析、广告类SDK

隐私政策	首次启动时弹窗展示，用户需点击“同意”后方可继续使用；若不同意，则退出App。

5.1.1 云端同步与账号体系详解

账号体系设计：



首次启动App时，生成匿名设备ID（UUID），所有本地数据与该ID关联，但不自动上传。



用户若开启“云同步”功能，首先提示绑定手机号（可选）。绑定流程：输入手机号 → 接收验证码 → 验证成功后，将匿名ID与手机号绑定，云端开始同步数据。



绑定后，用户可在其他设备上通过手机号+验证码登录，拉取云端数据。



未绑定手机号的用户，仍可使用云同步功能（如同设备换机时，通过本地备份文件恢复），但无法跨设备同步。



付费与账号强绑定：



所有付费功能（包括依从性分析、库存管理、数据导出、云端同步、完整事件驱动）必须基于已登录且已绑定手机号的账号才能使用。免费用户可继续使用v2.7的全部功能，无需登录。



用户通过应用内购买（IAP）支付成功后，付费权益与当前登录的账号永久绑定。换设备后，只要登录同一账号，即可恢复付费状态。



提供“恢复购买”按钮，用户可通过验证Apple/Google收据，将付费权益恢复到当前登录的账号。



数据迁移路径：



v2.7用户升级到v2.8后，首次打开App时，引导用户创建/登录账号，并提示将本地数据迁移到云端，确保数据不丢失。



用户可选择“稍后”跳过，但开启云端同步或使用付费功能时需强制登录。



隐私说明：



在用户绑定手机号及登录时，明确提示收集手机号仅用于账号恢复、数据同步及付费权益管理，不会用于其他用途。隐私政策中需增加相关说明。



5.1.2 完整事件驱动方案说明

交互流程：



用户在添加药品时选择“事件驱动”方案。

选择预设模板（如“2+1-1方案”）或自定义序列。

勾选预计有性行为/事件发生的日期。

系统自动计算并生成完整的提醒计划（包括事件前、事件中、事件后的所有提醒）。

提醒规则：支持首次双倍剂量、后续每日单剂量等复杂规则。



记录与统计：按事件统计依从性，而非按日统计。

6\. 技术方案要点（含扩展性设计）

6.1 前端框架

跨平台框架：React Native 或 Flutter（根据团队熟悉度选择）



本地数据库：SQLite + SQLCipher（通过对应框架的ORM或原生插件）



6.2 提醒调度

平台	实现方式

Android	AlarmManager（setExactAndAllowWhileIdle）+ BOOT\_COMPLETED广播恢复 + **自启动权限（为保证华为/小米等国内厂商的兼容性，需申请自启动权限）**

iOS	UNUserNotificationCenter + 本地通知

跨平台	优先使用成熟插件（如react-native-push-notification），评估后决定是否需要原生桥接

6.3 语音

系统TTS引擎，零成本。



6.4 数据加密

密钥由Android Keystore / iOS Keychain存储，不导出。



6.5 内置药品库

以JSON格式打包进App，首次启动时导入。



6.6 多天计划实现方案

生成方式：在添加时，根据用户输入的起始日期和持续天数，循环创建多条Medication记录，每条记录的schedule字段存储为一次性提醒（可用一个标记，如{"type":"one\_time","date":"2026-03-10","time":"08:00"}）。



分组关联：增加plan\_group\_id字段，将同一计划的多条记录关联，便于后续统一管理或删除。



模板存储：模板以JSON格式保存在Settings表中，用户可保存和加载。模板内容只包含“持续天数”和“每日剂量序列”，使用时需用户指定药品、起始日期、提醒时间。



6.7 停用机制实现

字段：isActive (Boolean, 默认true)，stoppedAt (Date, 默认null)。



查询：药箱活跃列表查询isActive=1，停用列表查询isActive=0。



提醒调度：仅扫描isActive=1的项目。



记录统计：records表关联medications时，不论isActive状态，所有记录均参与统计。



6.8 数据模型扩展预留（为v2.8准备）

当前v2.7版本所有表不包含user\_id字段，所有数据属于本地单用户。



v2.8升级时，需执行数据库迁移脚本，为Medication、Record、Settings表增加user\_id字段（类型为TEXT），并为现有记录赋予默认值（当前设备生成的匿名ID）。此迁移不影响现有数据。



匿名ID生成策略：首次启动App时，调用系统API生成唯一UUID，存储在Settings表中（key为device\_uuid）。该ID用于标识当前设备，v2.8绑定账号时作为过渡标识。



6.9 提醒规则引擎的可扩展性设计（为v2.8预留）

当前schedule字段使用JSON数组，示例：



json

\[

&nbsp; {

&nbsp;   "type": "daily",

&nbsp;   "time": "08:00"

&nbsp; }

]

为支持v2.8“完整事件驱动”，需预留扩展类型：



json

{

&nbsp; "type": "event\_driven",

&nbsp; "eventDates": \["2026-05-01", "2026-05-03"],

&nbsp; "protocol": "2+1-1",

&nbsp; "customSequence": \[

&nbsp;   {"dayOffset": 0, "dose": 2},

&nbsp;   {"dayOffset": 1, "dose": 1},

&nbsp;   {"dayOffset": 2, "dose": 1}

&nbsp; ]

}

开发要求：解析schedule字段时，对未知类型应忽略或报错，但数据库设计需能存储任意JSON结构，不做强校验。

## 10. 边界问题确认（2026-03-06补充）

### 10.1 数据库加密

- **要求**：v2.7 MVP必须实现SQLCipher加密
- **实现方案**：使用 `sqflite_sqlcipher` 包，密钥存储于 `flutter_secure_storage`（Android Keychain / iOS Keychain）
- **加密方式**：AES-256全库加密

### 10.2 Schedule提醒规则数据结构

- **存储格式**：JSON数组，字段类型为TEXT
- **结构规范**：
```json
[
  {"type": "daily", "time": "08:00"},
  {"type": "weekly", "time": "08:00", "days": [1, 3, 5]},
  {"type": "monthly", "time": "08:00", "dates": [1, 15]},
  {"type": "interval", "hours": 8},
  {"type": "one_time", "date": "2026-03-10", "time": "08:00"}
]
```
- **星期编号**：1=周一，7=周日
- **one_time规则**：必须包含完整日期（date字段）和时间（time字段）

### 10.3 多天计划结束后处理

- **决定**：保留记录
- 计划执行完毕后，相关记录标记为已完成，保留在数据库中供历史查看
- 不自动删除

### 10.4 过期提醒处理

- **决定**：不补发通知
- 用户打开App时显示"您有X天未确认"（如有），但不补发系统通知

### 10.5 连续添加模式

- **匹配精度**：精确到分钟
- "同一时间"指分钟也相同（如08:00和08:00可连续添加，08:00和08:30不可）

### 10.6 首次启动引导

- **决定**：欢迎页 + 隐私政策 + 用户协议
- 启动流程：
  1. **欢迎页面（Splash Screen）**：展示绿色渐变背景 + 医疗图标 + 标语"服药宝，守护您的用药健康"，2.5秒后自动跳转
  2. **用户协议 + 隐私政策**：用户需阅读并同意用户协议和隐私政策后，才能进入应用
  3. **通知权限引导**：同意协议后弹窗引导用户开启通知权限
- 欢迎页面每次启动都会显示（非首次启动也会显示）
- 用户不同意协议则退出应用

### 10.7 合并提醒数量限制

- **决定**：显示前3个
- 无论多少个药合并，通知标题和语音播报只显示前3个+ "等X种"
- 示例："请服用硝苯地平1片、鱼油1粒、维生素C1片等5种"

### 10.8 开发流程调整

- **UI设计优先**：先完成UI设计并确认后，再进行代码开发
- **UI设计内容**：四个模块（首页、药箱、记录、设置）的完整原型
